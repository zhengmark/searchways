# Multi-Agent Architecture for "现在就出发" Route Planning

> **实施状态**: ✅ 已完成 (2026-04-29) | 代码行数: ~800 行新增 Python
> **提交**: 202781e — 实现多智能体协同架构（Plan-Execute-Review-Refine）

## Context

当前系统是单流水线架构 (`agent/core.py` 1075 行)，存在几个问题可以通过多智能体解决：
- 意图解析是单次 LLM 调用，容易出 placeholder 或截断
- POI 搜索策略硬编码（4 点采样），缺乏对用户偏好的深度推理
- 路线规划后无质量审核，不合理的路线直接输出给用户
- 解说生成缺乏个性化（语气、详细程度无法适配老年/亲子等不同群体）
- 无法做"多方案对比"——用户只能看到一条路线

目标：引入多智能体协同，让路线规划从"单流水线"变成"多专家协作"，提升路线质量和用户体验。

## Architecture Overview

```
用户输入
  │
  ▼
┌──────────────────────────────────────────┐
│           Orchestrator (主控)             │
│  管理协作流程、工具执行、状态持久化          │
└──────────────────────────────────────────┘
  │           │            │            │
  ▼           ▼            ▼            ▼
┌────────┐ ┌────────┐ ┌────────┐ ┌──────────┐
│ Intent │ │  POI   │ │ Route  │ │ Quality  │
│ Agent  │ │Strategy│ │Narrator│ │ Reviewer │
│        │ │ Agent  │ │ Agent  │ │  Agent    │
│ 意图理解 │ │ 搜索策略 │ │ 路线解说 │ │ 质量审核   │
└────────┘ └────────┘ └────────┘ └──────────┘
                    │
              ┌─────┴──────┐
              │ Route Engine │  ← 算法引擎，非 LLM
              │ (复用现有)    │
              └────────────┘
```

### Agent 角色定义

| Agent | 职责 | 输入 | 输出 |
|-------|------|------|------|
| **Intent Agent** | 深度理解用户需求，提取结构化出行意图，补充隐含偏好 | 用户原始输入 + 城市 | `UserIntent` (含偏好推理、群体适配、季节/时间建议) |
| **POI Strategy Agent** | 制定搜索策略，评估搜索结果质量，决定是否需要补搜 | `UserIntent` + 起终点坐标 | 搜索计划 (区域/关键词/半径) + POI 质量评估 |
| **Route Narrator Agent** | 用个性化语言解说路线，适配用户群体特征 | 路径结果 + 用户画像 | 叙事文本 (语气/详细度可配置) |
| **Quality Reviewer Agent** | 审核路线合理性，发现问题时退回修正 | 完整路线方案 | 评分 + 问题清单 + 修正建议 |

### 协作流程 (Plan-Execute-Review-Refine)

```
Phase 1 ─ PLAN
  Orchestrator → Intent Agent: 深度解析用户意图
  Orchestrator → POI Strategy Agent: 基于意图制定搜索策略
  Orchestrator: 执行工具调用 (geocode, search_poi, 现有 tools/)
  Route Engine: 建图 + 最短路径 (纯算法，不经过 LLM)

Phase 2 ─ NARRATE
  Orchestrator → Route Narrator Agent: 根据用户画像解说路线

Phase 3 ─ REVIEW & REFINE
  Orchestrator → Quality Reviewer Agent: 审核路线质量
  if 有问题:
    Orchestrator → POI Strategy Agent: 根据反馈调整搜索
    重新执行 Route Engine → 再次审核 (最多 2 轮)

Phase 4 ─ OUTPUT
  最终输出: 解说文本 + Mermaid + Leaflet HTML
```

## Key Design Decisions

### 1. Agent 间通信: 结构化 Pydantic 消息

每个 Agent 是一个 async 函数，接收 Pydantic 对象，返回 Pydantic 对象。Orchestrator 负责序列化/反序列化和传递。不引入消息队列或复杂的事件总线——用函数调用即可。

```python
# agent/multi_agent/types.py 示例
class AgentMessage(BaseModel):
    agent_name: str
    task: str
    context: dict
    result: Optional[dict] = None

class IntentResult(AgentMessage):
    origin: str
    destination: Optional[str]
    keywords: list[str]
    num_stops: int
    user_profile: dict       # 新增: 推断的用户画像
    preference_reasoning: str # 新增: 偏好推理过程
    search_hints: list[str]   # 新增: 给 POI Agent 的搜索提示
```

### 2. 算法引擎不动

`graph_planner.build_graph()` + `shortest_path()` 保持为纯算法，不经过 LLM。这是项目中已验证正确的核心设计决策。

### 3. 最大 2 轮审核循环

Reviewer → POI Strategy → Route Engine → Reviewer 最多执行 2 轮，避免无限循环和 API 成本失控。

### 4. 复用小模型降本

- Intent Agent: 用 `LLM_MODEL` (当前 LongCat-Flash-Chat)，常规调用
- POI Strategy Agent: 用 `LLM_MODEL`，max_tokens 300
- Route Narrator: 用 `LLM_MODEL`，max_tokens 1500 (最长输出)
- Quality Reviewer: 用 `LLM_MODEL`，max_tokens 400 (轻量审核)

所有 Agent 共用同一个 API endpoint 和 key。

### 5. 向后兼容

`agent/core.py` 的 `run_agent()` 保留但标记 `@deprecated`。新增 `agent/multi_agent/orchestrator.py` 的 `run_multi_agent()` 作为新入口。`main.py` 改为调用新入口，但老测试仍可 import 旧函数。

## Implementation Steps (按依赖排序)

### Step 1: 定义 Agent 通信类型

File: `agent/multi_agent/types.py`

- 定义 `UserProfile` (群体类型/年龄偏好/体力等级/预算/兴趣标签)
- 定义 `IntentOutput` (扩展当前 UserIntent，加推理过程)
- 定义 `SearchStrategy` (搜索区域/关键词/半径/优先级)
- 定义 `PoiQualityReport` (覆盖率/多样性/评分分布)
- 定义 `ReviewResult` (评分卡 + 问题清单 + 是否需要重试)

### Step 2: 实现 Intent Agent

File: `agent/multi_agent/intent_agent.py`

- System prompt: 出行意图分析专家
- 输入: 用户原始输入 + 城市
- 输出: `IntentOutput` (含 origin/destination/keywords/num_stops + 用户画像推断)
- 新增能力: 推断隐含偏好 (如"带我妈"→偏好安静/低步行/有座位/无障碍)
- 新增能力: 输出 `search_hints` 给下游 POI Agent (如"优先江浙菜/茶楼/公园")
- 保留现有 `_parse_intent` 的占位符校验逻辑

### Step 3: 实现 POI Strategy Agent

File: `agent/multi_agent/poi_strategy_agent.py`

- System prompt: POI 搜索与评估专家  
- 输入: `IntentOutput` + 起终点坐标
- 输出: 搜索策略列表 + 搜索后的 POI 质量评估
- 新增能力: 决策是否需要补搜/调整半径/切换关键词
- 新增能力: 对搜索结果打分 (覆盖率/多样性/匹配度)
- 实际 API 调用仍由 Orchestrator 执行 (Agent 只做决策)

### Step 4: 实现 Route Narrator Agent

File: `agent/multi_agent/narrator_agent.py`

- System prompt: 出行路线解说员
- 输入: 路径结果 + 用户画像 + 城市
- 输出: 叙事文本
- 新增能力: 根据 `user_profile.group_type` 调整语气
  - family: 温馨/照顾老人小孩/强调安全舒适
  - couple: 浪漫/有氛围/拍照点
  - friends: 活泼/网红/性价比
  - solo: 自由/深度/探索感
- 保留现有解说 prompt 的核心约束 (不修改站点顺序和交通工具)

### Step 5: 实现 Quality Reviewer Agent

File: `agent/multi_agent/reviewer_agent.py`

- System prompt: 路线质量审核员
- 输入: 完整路线方案 (含站点/距离/时间/交通工具)
- 输出: `ReviewResult` (评分卡 1-5 + 问题清单 + retry flag)
- 检查维度:
  - 距离合理性: 单段步行 > 3km 且标记"步行" → 警告
  - POI 多样性: 3 个站全是同品类 (如全是火锅) → 建议替换
  - 总耗时: 超过用户时间预算 → 警告
  - 用户适配: 老年群体但有长距离步行段 → 建议调整
  - 空间分布: POI 扎堆在 500m 内 → 建议扩大搜索
- 返回 `needs_retry: bool`，触发时 Orchestrator 带着 Review 反馈重新调用 POI Strategy Agent

### Step 6: 实现 Orchestrator

File: `agent/multi_agent/orchestrator.py`

- 主函数: `run_multi_agent(user_input, session) -> (response, session)`
- 管理 Phase 1-4 流程
- 负责工具执行 (geocode, search_poi, search_around, build_graph, shortest_path)
- 管理审核循环 (最多 2 轮)
- 保持与现有 `AgentSession` 兼容

### Step 7: 更新入口

File: `main.py`
- 改为 `from agent.multi_agent.orchestrator import run_multi_agent`
- 其余逻辑不变 (CLI loop)

### Step 8: 扩展测试

File: `test_multi_agent.py` (新建)
- 3 个标准场景的端到端测试
- 新增 2 个需要多智能体协同的场景:
  - 极简输入 ("北京 吃" → 需要深度推理)
  - 多偏好 ("带老人和小孩去杭州玩半天，要能坐下来休息的地方")

## Risks & Mitigations

| Risk | Mitigation |
|------|-----------|
| API 调用次数 5-8x 增加 (4 个 Agent + 审核循环) | 每个 Agent 用较小的 max_tokens；审核循环限 2 轮；可加 `--fast` 模式跳过 Reviewer |
| 多 Agent 引入更多 LLM 不确定性 | 每个 Agent 输出严格的结构化 JSON + post-parse 校验，失败的 Agent 降级到现有逻辑 |
| 延迟增加 (串行调用 4 个 Agent) | Intent Agent 和 POI Strategy Agent 可并行 (仅依赖用户输入 + 城市)；后续可引入 streaming |
