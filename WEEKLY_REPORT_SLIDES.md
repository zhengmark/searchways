# 二、架构演进：为什么改

**原架构**：LLM 选 POI → 计算两两距离 → 人工排序

- LLM 对空间关系无感知
- 3 个站点全部扎堆在起点 37m 范围内
- "逛吃路线"变成"门口转一圈然后打车走人"

**新架构**：起终点 + POI 建模为加权图

```
意图解析 → 地理编码 → 走廊POI搜索 → 建图+路径规划 → 解说 → 输出
```

---

# 架构演进：路径算法迭代

**初版 — 贪心最近邻**（已废弃）

从起点出发，每次选最近的未访问 POI → 所有 POI 仍扎堆起点

**终版 — 连线投影 + 分段选取**

POI 投影到起点→终点连线上，均分 N 段后每段选最优，保证空间自然分散、无回溯

# 深度修复：地名截断

"丈八四路地铁站" → LLM 截断为 "四路地铁站" → 误匹配到凤城四路

**4 层防御体系**

| 层 | 策略 |
|:--|------|
| 1 | LLM prompt 强化，禁止截断 |
| 2 | `robust_geocode()` 4 层回退 |
| 3 | 正则从用户原始输入回退 |
| 4 | 连续子串校验防跨地名误配 |

> 防御核心在层 1——截断了且 input_tips 不含正确项，则完全不可恢复

# 六、关键技术决策 (1/2)

| 决策 | 选择 | 放弃原因 |
|------|------|------|
| 路径算法 | 连线投影 + 分段选取 | TSP 是 NP-hard，贪心近似足够 |
| 交通工具估算 | 步行 API × 速度系数 | 公交/骑行 API 覆盖率低，成本高 |
| POI 搜索 | 4 点采样 | 全路径采样 API 调用 O(n) 爆炸 |

---

# 关键技术决策 (2/2)

| 决策 | 选择 | 放弃原因 |
|------|------|------|
| LLM 角色 | 意图解析 + 解说 | LLM 无空间感知，选点质量不可控 |
| 坐标检查 | `is not None` 显式 | `if value` 误杀 `0.0` 有效值 |
| 地名匹配 | 连续子串 | 字符重叠率跨地名时易误匹配 |

---

# 七、关键认知收获

1. **不信任 LLM，信任验证** — JSON 当可信输入是死路，层层校验才稳定
2. **兜底需要递进设计** — 精确 API → 模糊 API → 数学估算 → 正则回退
3. **空间规划 ≠ 最短路径** — 线段投影 + 分带取优更接近人类逻辑
4. **低信息量是试金石** — "北京 吃"能跑但路线短，是管线边界清晰的表现
