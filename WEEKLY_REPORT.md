## 二、架构演进：从 "POI 排序" 到 "图上路径规划"

### 为什么改

原架构：LLM 选 POI → 计算两两距离 → 人工排序。核心问题是 **LLM 对空间关系没有感知**，选出的 3 个站点可能全部扎堆在起点 37m 范围内，然后一个大跳跃到终点——"逛吃路线"变成"门口转一圈然后打车走人"。

### 怎么改

将起终点 + POI 建模为加权图（邻接矩阵），每条边的权重由高德步行 API 实测距离填充（失败时 haversine 估算兜底），在图上用算法计算最优路径。

```
LLM解析意图 → 地理编码(4层兜底) → 走廊POI搜索 → 建图+路径规划 → LLM解说 → Mermaid+HTML输出
```

### 路径规划算法的两次迭代

**初版 — 贪心最近邻**（已废弃）

从起点出发，每次选距当前节点最近的未访问 POI，加前进方向奖励和多样性惩罚。问题：所有 POI 仍扎堆起点——`search_along_route` 搜到的 POI 高度集中在起点周边，最近邻即使加权也跳不出局部最优。

**终版 — 起终点连线投影 + 分段选取**

将每个 POI 投影到起点→终点的地理连线上，得到 [0,1] 的投影比例，均分 N 段后每段选评分最高的 POI，保证空间自然分散、无回溯。无终点场景改用"距起点距离分层"，同样均分 N 个距离带，并用 200m 最小间距约束避免同楼扎堆。

## 四、深度修复：地名截断问题

用户输入 "丈八四路地铁站" → LLM 截断为 "四路地铁站" → geocode 误匹配到凤城四路（城市另一端）。

**根因链**：LLM 截断多字地名 → input_tips 的字符重叠校验用 `overlap >= threshold`，但 "四路""地铁站" 拆开分散在不同地名中也能达标，造成跨地名误匹配。

**4 层防御体系**：

| 层 | 策略 | 作用 |
|----|------|------|
| 1 | LLM prompt 强化（禁止截断 + 正例示范） | 阻止源头 |
| 2 | `robust_geocode()` 4 层回退（原始名称 → input_tips → 加后缀 → 城市前缀） | 模糊修复 |
| 3 | 正则从用户原始输入回退提取地名模式 | 绕过 LLM 错误 |
| 4 | input_tips 匹配改为连续子串校验（`name in tip_name`） | 防跨地名误配 |

**关键认知**：防御的核心在层 1——阻止 LLM 截断——而非事后修复。如果 LLM 截断了 "丈八" 且 input_tips 不返回含 "丈八四路" 的候选项，则完全不可恢复。

## 六、技术决策记录

| 决策 | 选择 | 替代方案及放弃原因 |
|------|------|-------------------|
| 路径算法 | 连线投影 + 分段选取 | Dijkstra 全源最短路：边权重基于步行 API 非欧几里得距离；TSP 变种是 NP-hard，贪心近似足够 |
| 交通工具估算 | 步行 API × 速度系数 | 接入公交/骑行 API：覆盖率低，高德公交 API 需额外 Key，成本太高 |
| POI 搜索 | `search_along_route` 4 点采样 | 完整路径点采样：API 调用量 O(n) 爆炸，4 点采样在 <20km 路线中足够 |
| LLM 角色 | 意图解析 + 路线解说 | 路径决策交给 LLM：LLM 无空间感知，选点质量不可控 |
| Python falsy | `is not None` 显式检查 | `if value` 惯用法：坐标 0.0 会误杀有效数据 |
| 地名匹配 | 连续子串（`a in b`） | 字符重叠率：跨地名时易误匹配，如"四路地铁站"散落在"凤城四路 XXX 地铁站 XX" |

## 七、关键认知收获

1. **不信任 LLM，信任验证**：LLM 返回 JSON 就当作可信输入是死路。占位符校验、子串校验、坐标真实性检查层层把关，才是系统稳定的基础。

2. **兜底需要递进设计**：每一处外部调用都可能失败。从精确 API → 模糊 API → 数学估算 → 原始输入正则回退，每一层的失败都比上一层更安静。

3. **空间规划 ≠ 最短路径**：纯图论最短路把所有 POI 挤在起点。线段投影 + 分带取优本质上是在做「空间均匀性」约束下的贪心选择，更接近人类规划路线的真实逻辑。

4. **低信息量输入是系统的试金石**："北京 吃"能跑出东西但路线短（1.4km），"深夜小吃"诚实告诉你只找到 1 个。这是管线边界清晰的表现，而非缺陷。
